**Function calling：在我这个系统里，“Function Calling 多轮循环的 6 轮”，指的是一次用户提问里，最多和大模型往返 6 次：每一轮我先调一次大模型，它决定要不要调用哪些工具，我在本地执行工具，把结果再喂回去；如果还需要信息就进入下一轮。也就是说，是“最多 6 次 LLM 调用 + 每轮若干工具调用”，而不是“只限制 6 次工具调用。每一轮调用大模型的时候，大模型可以提出使用多个工具的需求，也就意味着一次可以使用多个工具。**

**1、在你这个项目里，大模型能调用的“工具”主要分两类：一类是你自己在本地实现并显式声明给它的工具，另一类是模型服务商在云端已经帮你“接好”的内置工具接口（比如官方的 web 搜索等）。**
你现在在项目中调用的工具，绝大部分确实是自己实现的业务工具（PDF 检索、引用定位、代码执行等）。大模型本身只“自带”极少量、很通用的工具能力，而且通常也要你在接口里显式启用或声明，需要通过 JSON Schema 暴露出来的业务工具，它才会用，比如官方的 web 搜索 / 浏览器 / 通用 code interpreter 等。那些你以为“模型自带的高级 tool”（复杂搜索、RAG、特定业务逻辑等），确实都需要你另外接入/实现，然后通过 Function Calling 暴露，模型才可能调用——它不会凭空知道你有 Chroma、有 LangGraph、有 PM2.5 模型。

**2、大模型本身只提供“语言+推理”的基础能力。**
即便你已经接入了大模型 API，它“原生”能做的就是理解问题、规划步骤、决定什么时候用工具，而不是自动拥有复杂业务功能。所谓“自带的 tool”，一般也就少量非常通用的东西（比如某些平台的浏览器、简单代码执行等），而且通常要显式启用。所有复杂功能都要通过“工具”接到它身边工具可以是你自己写的（检索、读 PDF、查数据库、算指标等），也可以是模型厂商或第三方写好的服务（比如官方 web 搜索、官方 code interpreter）。但不管谁写的，想让大模型在对话里“感知并调用它”，都要按统一方式暴露：
给工具起名字、写描述；定义参数 JSON Schema（告诉模型要传什么字段）；在调用 API 时，把这些定义放进 tools / functions 里；
只有这样，模型才知道：“哦，我现在有这些按钮可以按，这个按钮能干什么，怎么填参数”。

Function Calling 多轮循环 = 让模型反复“用工具 → 看结果 → 再决策”
每一轮流程是：你把“对话历史 + 之前工具结果”发给大模型；它决定：当前这轮要不要用某个工具，用哪个，用什么参数；你在后端真实执行工具；把工具结果再塞回对话
这样最多可以重复多轮，让模型像 Agent 一样一边探索、一边用工具补信息。
你现在的系统确实设了“最多 6 轮工具循环 + 失败回退机制”；最多 6 轮 Function Calling 往返，如果在 6 轮之内模型还没给出最终回答，就触发“回退策略”：直接走一次性 RAG/单轮问答模式，简单粗暴地给一个答案，而不是继续死循环。
这既是为了控制成本（最多调用 6 次 LLM），也是为了防止“模型一直纠结、来回查却不回答”。

**3、在你这套 RAG/Agent 系统里，就你现在这套项目来说，大模型能用的工具几乎都是你本地写好的：**
RAG 里的 4 个工具：retrieve_documents、get_citation_details、extract_formula_or_table、code_execution，都是你在后端 Python 里实现，然后通过 tools schema 暴露给 Anthropic 的。

Agent 模式里的 document_search、web_search、calculator、安全 code_executor 也是你在本地或者自己服务器上实现，再通过 LangChain / LangGraph 包一层给模型调用。

换句话说：

“工具” = 你写的函数 + 你定义的 JSON Schema + 你在对话里声明给模型，它才知道有这些工具可用。

模型本身不会凭空知道你的 Chroma、你的代码执行器，它只会按照你给出的工具列表来“点按钮”。

**4、模型有没有“自带工具”？**
从接口的角度看：
有些平台会提供“内置工具”，比如：官方的 browser / internet_search / code_interpreter 等，这些是在服务商后端实现好的，你只需在 tools 里声明它们的名字和参数格式，就能用。但本质上，对模型来说，这些也还是“外部工具”：它只是在输出里写出“我要调用某个名字的 tool”，真正执行仍然是平台的后端在跑。

对你自己的工程来说：你现在用的 Anthropic SDK + 自己写的工具，就是“自定义工具为主”。如果以后接入类似“官方 web 搜索 / 官方代码执行器”，那就是“部分工具由模型服务商提供，部分由你本地提供”，但从模型视角都是一样的：都是在 tools 列表里声明出来，它再选择调用。

所以，可以记住一句话：
**工具从来不是模型“脑子里自带的”，而是“谁在后端实现并在接口里声明，模型就能用谁”；你现在系统里的工具都是你本地写好并暴露给模型的，官方“自带”的其实只是另一批已经帮你实现好的外部工具而已。**

Function Calling 多轮工具调用，可以看成是 Agent 的“发动机”，负责让大模型按需调用工具、读结果、再决策；而 Agent 则是在这个基础上，加上状态管理、任务规划、记忆和错误恢复等能力，变成一个长期运行、可编排的智能体。我的项目里，RAG 模式更多是“用 Function Calling 做智能检索”，Agent 模式才是基于 LangGraph 的完整智能体工作流。
