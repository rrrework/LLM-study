**在我这个系统里，“Function Calling 多轮循环的 6 轮”，指的是一次用户提问里，最多和大模型往返 6 次：每一轮我先调一次大模型，它决定要不要调用哪些工具，我在本地执行工具，把结果再喂回去；如果还需要信息就进入下一轮。也就是说，是“最多 6 次 LLM 调用 + 每轮若干工具调用”，而不是“只限制 6 次工具调用。每一轮调用大模型的时候，大模型可以提出使用多个工具的需求，也就意味着一次可以使用多个工具。**

**在你这个项目里，大模型能调用的“工具”主要分两类：一类是你自己在本地实现并显式声明给它的工具，另一类是模型服务商在云端已经帮你“接好”的内置工具接口（比如官方的 web 搜索等）。**

**1. 在你这套 RAG/Agent 系统里，就你现在这套项目来说，大模型能用的工具几乎都是你本地写好的：**
RAG 里的 4 个工具：retrieve_documents、get_citation_details、extract_formula_or_table、code_execution，都是你在后端 Python 里实现，然后通过 tools schema 暴露给 Anthropic 的。

Agent 模式里的 document_search、web_search、calculator、安全 code_executor 也是你在本地或者自己服务器上实现，再通过 LangChain / LangGraph 包一层给模型调用。

换句话说：

“工具” = 你写的函数 + 你定义的 JSON Schema + 你在对话里声明给模型，它才知道有这些工具可用。

模型本身不会凭空知道你的 Chroma、你的代码执行器，它只会按照你给出的工具列表来“点按钮”。

**2. 模型有没有“自带工具”？**
从接口的角度看：
有些平台会提供“内置工具”，比如：官方的 browser / internet_search / code_interpreter 等，这些是在服务商后端实现好的，你只需在 tools 里声明它们的名字和参数格式，就能用。但本质上，对模型来说，这些也还是“外部工具”：它只是在输出里写出“我要调用某个名字的 tool”，真正执行仍然是平台的后端在跑。

对你自己的工程来说：你现在用的 Anthropic SDK + 自己写的工具，就是“自定义工具为主”。如果以后接入类似“官方 web 搜索 / 官方代码执行器”，那就是“部分工具由模型服务商提供，部分由你本地提供”，但从模型视角都是一样的：都是在 tools 列表里声明出来，它再选择调用。

所以，可以记住一句话：
**工具从来不是模型“脑子里自带的”，而是“谁在后端实现并在接口里声明，模型就能用谁”；你现在系统里的工具都是你本地写好并暴露给模型的，官方“自带”的其实只是另一批已经帮你实现好的外部工具而已。**
