# 5、你刚才说你重点优化了 chunk 策略：当时你具体怎么切分的（规则、目标 chunk 长度/Token 范围、overlap 大概多少），为什么这样选？相比“固定长度切分”你有做过什么可量化的对比结果或观察到的改进吗？
当时我优化 chunk 策略，核心出发点是：科研资料不是普通连续文本，它有很强的结构性，比如章节、子章节、定义、公式、图表说明这些内容，如果直接固定长度切分，很容易把一个完整语义单元切断，导致召回到的片段“字面相关但信息不完整”，最后影响生成质量。这个项目里我简历里也写了，主要是按章节、小节和公式段落来做文档切分。
具体做法上，我当时采用的是一种结构优先、长度约束兜底的切分方式。
第一层先识别文档结构，比如标题层级、段落边界、公式块、列表这些，尽量保证一个 chunk 内部是相对完整的语义单元；
第二层再做长度控制，避免 chunk 过长影响 embedding 质量，或者过短导致上下文不足。

在长度上，我当时控制的思路大概是：普通正文 chunk 尽量落在几百 token这个区间；太短的相邻小段会适当合并；太长的章节段会再往下切。
如果要给一个更具体的范围，可以说我当时目标是大致控制在 300 到 500 token 左右，overlap 一般会保留 大约 50 到 100 token，主要是为了减少跨段切分导致的信息断裂，尤其是一些定义和解释分散在相邻段落时，overlap 能提升召回连续性。

我这样选的原因主要有三个：第一，科研资料里局部上下文非常重要。比如一个公式本身没意义，通常要结合前后的符号定义、变量解释和结论描述来看，所以不能切得太碎。
第二，embedding 检索更适合语义完整、但又不过长的片段。太短会导致语义信息不足，召回结果容易飘；太长又会把多个主题混在一起，降低检索精度。
第三，生成侧也需要可控上下文。如果每个 chunk 太大，送进 prompt 时容易挤占上下文窗口；太小又需要拼很多片段，反而增加噪声。
相比固定长度切分，我当时观察到两个比较明显的改进：
一个是召回结果相关性更稳定。固定长度切分时，经常会出现“问题明明和某个知识点有关，但召回出来的 chunk 正好只包含前半段背景，或者只包含后半段结论”，信息不闭环。改成结构化切分后，这种情况会少很多。
第二个是最终回答的一致性和引用可读性更好。因为 chunk 更接近自然段落和章节单元，所以模型生成答案时更容易基于完整证据作答，返回的引用片段也更像人能直接阅读和验证的内容，而不是一截被截断的文本。简历里提到的“引用返回”和“相关性提升”，本质上也和这个优化有关。
如果说量化对比，我当时更偏工程验证，不是做特别严格的 benchmark。所以我主要是通过检索 Top-K 结果人工抽样对比，以及看最终问答的相关性、完整性、引用准确性来判断。我的观察是，结构化切分相比纯固定长度切分，在科研文档这种强结构文本上，前几条召回结果会更聚焦，回答里“答到点上但证据不完整”的情况会明显减少。

如果让我把结构化切分和固定长度切分做成可复现的离线评测，我会先基于那批科研 PDF 构造一个问答评测集，每个问题不仅有问题文本，还会人工标注对应的 gold evidence，也就是标准证据段落。然后实验里尽量只改 chunk 策略，其他都固定，比如文档、清洗流程、embedding、向量库、召回参数、rerank 和生成模型都不变。
对照组就是固定长度切分，实验组是按章节、小节、公式段落这些结构边界优先切分，再做长度约束。最终我会优先看检索指标，核心是 Recall@K 和正确证据排序，比如 MRR 或 nDCG；另外我还会补一个“证据完整性”指标，因为 chunk 策略最大的差异，往往不是有没有召回到，而是召回到的片段是不是完整、能不能直接支撑答案。只要结构化切分在召回率、排序和证据完整性上都更好，我就会认为它优于固定长度切分。

问答层评测集：问题 + 参考答案 + gold evidence
这样既能测召回，也能测最终回答。

# 7、你刚才提到“证据完整性”这个指标。你会把它怎么定义成可自动计算、而且不同标注者也能尽量一致的度量？（可以假设你已经有 gold evidence 的标注粒度，比如按段落/句子/页码。）
如果要把“证据完整性”定义成可自动计算的指标，我不会直接让标注者主观判断“这个 chunk 完不完整”，而是会基于已标注的 gold evidence 基础单元来算覆盖率。比如 gold 已经按句子或段落标好了，那我就把完整性先定义成召回结果对 gold 单元的覆盖比例，也就是 Evidence Coverage。进一步我会给核心单元加权，比如公式句、定义句、变量解释句权重大于辅助背景句，这样可以得到 Weighted Coverage。
另外我还会补一个更严格的 Complete Evidence Rate，就是先定义每种题型的必需证据角色，比如公式解释题至少要覆盖公式本身和变量解释，只有这些必需角色都被召回，才算证据完整。这样做的好处是它可以自动算，而且标注者只需要在 gold 标注时按统一粒度和角色标签去标，主观性会比直接打“完整/不完整”低很多，一致性也会更高。

# 8、你说的“必需证据角色”如果要落到一套可执行的标注规范上：你会定义哪些角色集合（比如定义/公式/变量解释/结论/实验设置等），以及标注者在什么规则下判断“这个句子/段落属于哪个角色”？（给我一个你最常见的题型示例也行。）
如果要把“必需证据角色”落成标注规范，我会先定义一套按证据功能划分的角色集合，而不是按句子表面形式划分。比如科研资料里我会常用 Claim/Definition、Formula、Parameter Explanation、Procedure、Constraint、Result、Comparison、Experiment Setup 这些角色。标注时的核心规则是看一句话在回答里承担什么功能，比如它是在直接给定义、给公式本体、解释变量、限定适用条件，还是在报告实验结果。
以最常见的公式解释题为例，如果文档里一段写公式本身，一段解释各个变量，还有一段说明这个公式为什么这么设计，我会分别标成 Formula、Parameter Explanation 和 Rationale。然后把这个题型的必需角色定义成 Formula 加 Parameter Explanation。这样后面做完整性评测时，只召回公式或者只召回变量解释都不算完整，必须核心角色都覆盖到才算完整。这套规则的好处是主观性相对低，而且标注者更容易达成一致。

1. 我会先**定义一套通用角色集合**
我的场景是科研资料问答，我先定义了一套比较稳定、覆盖大多数题型的角色集合：
A. 核心内容类
核心定义、命题、结论、方法主张（Claim / Definition）
判断标准：这句话本身能直接回答“是什么”“提出了什么”“结论是什么”。
公式本体、数学表达式、符号关系式（Formula）
判断标准：包含公式或等价数学表达，是后续解释的对象本身。
方法流程、步骤描述、算法过程（Procedure / Method Step）
判断标准：回答“怎么做”“流程是什么”“先后步骤是什么”。
B. 解释支撑类
变量、符号、参数含义说明（Parameter / Variable Explanation）
判断标准：句子主要在解释公式中某个符号、输入输出、变量物理意义。
适用条件、前提、边界、假设（Constraint / Condition）
判断标准：如果缺了这句，答案会变得过宽或不成立。
原理解释、为什么这样设计、对结果或方法的解释（Rationale / Interpretation）
判断标准：这句不是主结论本身，但在解释主结论“为什么成立”。
C. 结果验证类
数据集、实验环境、评价设置、对比设置（Experiment Setup）
判断标准：主要在说明结果是在什么实验条件下得到的。
定量结果、实验现象、性能表现（Result）
判断标准：包含指标、对比结果、性能提升或结论性实验观察。
与基线、其他方法或其他设定的对比（Comparison）
判断标准：句子核心作用是横向比较，而不是单独陈述本方法结果。
D. 辅助背景类
背景动机、领域现状、一般性描述（Background / Context）
判断标准：和问题相关，但通常不是回答该问题的必需证据。
2. 标注规则怎么定
我会要求标注者按“主功能优先”原则给句子或段落打标签。也就是说，一个句子可能同时有多种信息，但标注时优先看它**最主要在完成什么功能**。否则大家很容易一个句子打 3 个标签，最后一致性会很差。
规则 1：优先标“直接回答作用”
如果一句话本身已经能直接回答问题，就优先标成核心角色，比如 Claim、Definition、Result，而不是标成解释类。
规则 2：公式和公式解释分开
哪怕公式和变量说明挨在一起，也尽量区分：数学表达本身标 Formula，“其中 x 表示…” 这类标 Parameter / Variable Explanation；因为后续做完整性时，公式题通常需要这两个角色同时命中。
规则 3：条件和结论分开
像“在数据稀疏情况下，该方法效果更明显”这类句子，如果重点是限定适用范围，就标 Constraint；如果重点是陈述性能提升，就标 Result。判断关键看：去掉这句话后，缺的是“范围”还是“结论”。
规则 4：背景不算必需证据，除非问题本身问背景
比如“为什么要做这个研究”时，Background 才可能成为必需角色。如果问题问的是“方法是什么”或“公式含义是什么”，背景句即使相关，也通常不算完整性里的必需证据。
规则 5：按最小充分单元标
尽量在句子级或短段落级标，不把多个角色粗暴合并成一个大段。
因为一旦一个段落里既有公式又有变量解释又有结论，后面自动算 coverage 就很难精确。
4. 我会**怎么定义“题型—必需角色”映射**
为了让完整性评测更可执行，我不会只定义角色集合，还会定义不同题型必须覆盖哪些角色。
比如：
题型一：定义类问题
示例：
“什么是 xx 模型？”
“xx 指标的定义是什么？”
必需角色：Claim / Definition
可选角色：Constraint / Background
因为定义题最核心的是定义本身，背景不是必需。
题型二：公式解释类问题
示例：
“这个损失函数是什么意思？”
“公式中的各个变量代表什么？”
必需角色：Formula / Parameter / Variable Explanation
可选角色：Rationale / Interpretation / Constraint
这是最常见、也最能体现结构化切分价值的题型。如果只召回公式没召回变量解释，或者只召回变量解释没召回公式，我都不会算“证据完整”。
题型三：方法流程类问题
示例：
“这个方法具体怎么做？”
“模型训练流程是什么？”
必需角色：Procedure / Method Step / Claim（如果首句总结方法目标）
可选角色：Rationale / Constraint
因为流程题要能把步骤链条讲清楚，单独一个结论句不够。
题型四：实验结果类问题
示例：
“这个方法最终效果怎么样？”
“相比 baseline 提升了多少？”
必需角色：Result / Comparison
可选角色：Experiment Setup
如果问题问的是“提升了多少”，那没有 Comparison 就不完整；如果问“在什么数据集上验证的”，那 Experiment Setup 就会变成必需。
4. 给你一个最常见题型示例：公式解释题
我会在面试里重点讲这个，因为它最容易说明规则。
比如文档里有这样三段：
“损失函数定义为：L = Lcls + λLreg”
“其中 Lcls 表示分类损失，Lreg 表示回归损失，λ 为平衡系数”
“该设计能够同时约束分类能力和定位精度，在目标检测任务中更稳定”
那如果问题是：“这个损失函数是什么意思？”
我会这样标：
第 1 段：Formula
第 2 段：Parameter / Variable Explanation
第 3 段：Rationale / Interpretation
然后我会规定这个题型的必需角色是：Formula / Parameter / Variable Explanation 。第 3 段是加分信息，但不是严格必需。因为如果没有第 3 段，用户还是能知道公式是什么、变量代表什么；但如果没有前两者之一，答案就不成立。
这样后面做“证据完整性”时，就能自动判断：只召回第 1 段：不完整；只召回第 2 段：不完整；同时召回第 1、2 段：完整；1、2、3 都召回：完整且更充分
5. 怎么提高标注一致性
我会专门在规范里加几条裁决规则：裁决规则 A：按“回答缺失”判断角色
问自己：如果删掉这句话，答案会缺什么？
缺定义本体 → Claim / Definition
缺公式对象 → Formula
缺符号含义 → Parameter
缺适用范围 → Constraint
缺性能数据 → Result
这个判断比“读起来像不像定义句”更稳定。
裁决规则 B：一个单元默认一个主标签
除非确实无法拆开，否则一句话或一个最小段落只给一个主标签。这样能显著减少多标签造成的不一致。
裁决规则 C：先判题型，再判角色
先明确当前问题属于定义题、公式解释题、流程题还是结果题。
题型确定后，标注者对“哪些角色是必需的”会更一致。

# 9、你这个 RAG 知识库的离线构建流程里，“PDF 文本抽取 + 清洗”具体怎么做的：你用的是什么工具/库把 PDF 变成结构化文本的（尤其是公式、表格、页眉页脚、参考文献），以及你定了哪些清洗规则来避免把噪声（比如页码、脚注、目录）灌进向量库？
我这个 RAG 知识库离线构建，核心流程是 PDF 解析 → 文本清洗 → 结构恢复 → 分块和元数据标注 → 向量化入库。项目目标不是把 PDF 版式 100% 还原，而是把科研资料变成适合检索的结构化文本，因为后面要做召回、重排和引用返回。
在 PDF 文本抽取上，我当时用的是 Python 侧的 PDF 解析工具 去按页提取文本，再结合版面和规则恢复章节、段落这些结构。这里我的重点不是做特别重的版面理解，而是优先保留对检索有价值的信息，比如：章节标题、段落正文、公式附近的解释文本、表格标题或表头、页码和文档来源这类元数据
1. 公式怎么处理：公式这块我做的是实用导向处理，不是高精度公式还原。因为科研 PDF 里复杂公式经常抽取后会有乱码、断裂或者符号丢失，所以我不会强行把所有公式都恢复成标准数学表达，而是优先保证它在问答场景里的可用性：
能正常抽出来的公式，就和前后的变量解释、段落说明一起保留；
抽取得不理想的公式，不单独把一串噪声塞进向量库，而是重点保留它所在段的解释文本、公式编号、变量说明和上下文
在后续切分时，尽量把“公式 + 变量解释 + 局部说明”看成一个相对完整的语义单元
因为对 RAG 来说，很多时候比起公式字符串本身，更重要的是这个公式是干什么的、变量代表什么、适用在哪个方法里。
2. 表格怎么处理：表格也是类似思路。
规则表格如果抽取得比较干净，我会保留表头和主要行内容；如果表格抽出来已经明显错列、错行、碎片化比较严重，我不会把它原样灌进知识库，而是做降级处理，优先保留：
表格标题、表头、表格上下文描述、文中对表格结论的解释
因为乱掉的表格文本很容易污染 embedding，召回时反而会带来伪相关。
3. 页眉页脚、页码、脚注这些怎么处理
清洗上我主要做了几类规则，目标就是提升知识库信噪比，避免把噪声送进向量库。
第一类是跨页重复内容过滤。像页眉、页脚、页码、期刊抬头、版权声明这类内容，通常跨很多页重复，而且位置比较固定。我会基于“高频重复 + 固定模式”把这些内容清掉。
第二类是目录过滤。目录页看起来很像正文，因为会有很多标题，但它本质上没有可回答问题的知识内容，还容易误召回。所以我会根据这些特征过滤：
出现在前几页、大量“标题 + 点线 + 页码”的模式、命中“目录 / Contents”这类关键词
第三类是参考文献区处理。
如果我的目标是科研问答，而不是文献管理，那参考文献列表通常不会直接进主知识库。因为它包含大量作者名、年份、题目、期刊名，会稀释检索空间。所以我一般会在命中“参考文献 / References”后，对后面的纯文献列表做过滤或至少降权。
第四类是脚注和短噪声片段过滤。
像基金项目说明、通信作者信息、孤立编号、页脚注释、很短的碎片字符串，这些通常会根据：长度阈值、特定关键词、是否与正文语义连续来决定过滤。
第五类是断行合并和段落修复。
PDF 抽文本很容易一行一断，所以我会按标点、句子连续性和标题模式去做段落恢复，尽量让后面的 chunk 不要建立在碎裂文本上。
4. 我当时定的清洗规则可以概括成这几条：
去掉页眉、页脚、页码、版权抬头等跨页重复内容
过滤目录页和目录行
过滤或降权参考文献区
去掉过短、无语义的碎片文本和脚注
合并断行，恢复自然段和标题层级
对公式、表格附近说明文本做绑定，避免解释和对象本体分离
保留章节、页码、文档来源等元数据，方便后续引用定位
5. 为什么这些规则对 RAG 很重要

因为向量库不是“文本越多越好”，而是“有效信息密度越高越好”。
如果把页码、目录、页眉页脚、参考文献这些噪声直接灌进去，会有两个问题：

embedding 空间里会混入大量无意义相似项
用户提问时，Top-K 召回容易被伪相关内容干扰

所以 PDF 抽取和清洗，本质上是在做知识库的信噪比优化，这会直接影响后面的召回质量和回答可用性。这个项目里我后面做 chunk 策略、引用返回和问答优化，其实都建立在这一步的质量之上。


# 10、你离线解析 PDF 具体用的哪套库/工具链（比如 pdfplumber、PyMuPDF、GROBID 之类）？请你说清楚你在代码里实际走的流程：从 PDF 到“带章节/段落信息的结构化文本”，中间关键一步你是怎么拿到或推断出标题层级的。

我当时离线解析 PDF，走的是一条比较轻量但可控的 Python 工具链，不是走 GROBID 那种偏重学术论文解析的路线。
整体流程是：先逐页解析 PDF，提取文本块和版面信息，再做噪声清洗、段落恢复、标题层级识别，最后组织成带章节路径的结构化文本，再进入后续的分块和入库流程，这也对应我简历里写的 PDF 文本抽取、清洗、结构化分块和元数据标注。
具体来说，我不是一开始就把整篇 PDF 粗暴抽成纯文本，而是先按页拿到更细粒度的文本块、行或者 span，同时保留一些版面特征，比如页码、字体大小和粗细、文本在页面中的位置、行间距和段间距、是否独占一行、是否带有 1、1.1、2.3.1 这类编号模式，因为这些信息对后面区分标题和正文很关键。
拿到原始文本块后，我会先做一轮清洗，重点去掉页码、页眉页脚、目录、参考文献区以及过短的碎片文本，然后再做断行合并和段落恢复。这里我的原则是先把明显噪声清掉，再做结构恢复，否则目录行、页脚信息这类内容很容易干扰标题识别。
标题层级这一步主要靠规则推断，不是说 PDF 天然就能给出一棵目录树。我会综合几类信号去判断一个文本块是不是标题，以及它大概属于哪一级：
第一类是编号模式，比如 1、1.1、1.1.1、一、（一）这类，其中 1 或一、通常更像一级标题，1.1 或（一）更像二级标题，1.1.1 更像三级标题；
第二类是版面样式，比如字体更大、加粗、独占一行、前后留白更明显、出现在段落开头位置；
第三类是文本长度和句式特征，因为标题通常比较短，不像完整陈述句，而正文往往更长、带完整标点和连续表达。所以我会做一个综合判断：如果一个文本块既满足编号模式，又在版面上像标题，基本就可以比较稳地识别成某一级标题；如果没有显式编号，就更多依赖字体、位置和独占一行这些特征去判断它是不是无编号标题。识别出标题后，我会维护一个当前章节栈，比如当前一级标题、二级标题、三级标题，正文段落解析出来后就挂到最近的标题路径下面。
最后每个文本单元都会带上文档名、页码范围、一级标题、二级标题、三级标题、段落文本，以及是否包含公式或表格附近说明这类结构化元数据。这样后面做 chunk 的时候就不是盲切，而是可以按“章节—小节—段落”去组织，你简历里写的“按章节、小节和公式段落切分文档”，本质上就是建立在这一步结构恢复的基础上。
之所以没有用特别重的解析方案，是因为我这个项目的重点是验证科研资料问答场景下的 RAG 主链路，而不是做一个论文级 PDF parser，所以我更关注的是标题层级能不能大体恢复出来、段落能不能尽量完整、公式和解释有没有被拆散、噪声有没有被过滤掉。只要这些关键点处理好，后面的检索效果和引用返回稳定性就会明显更好。

# 12、RAG 里向量检索阶段你常用的相似度度量有哪些（比如 cosine / dot / L2），它们各自更适合什么前提（尤其是 embedding 是否做归一化）？结合你自己的知识库场景，说一下你选哪一个、为什么。
结合我自己的知识库场景，我会更倾向于选 cosine similarity，或者等价地先做归一化再用 inner product。因为我的项目本质上是科研资料问答，文档里有论文、教材、笔记这类内容，文本长度、表达风格、章节结构差异都比较大，而我更关心的是“问题和某个 chunk 在语义上是否匹配”，而不是哪个向量模长更大。我的项目里也做了 PDF 文本抽取、清洗、结构化分块、向量化存储和后续检索问答链路，所以对这个场景来说，稳定的语义匹配比保留模长信息更重要。
如果面试官再追问“那你为什么不直接说 dot product”，你可以补一句：如果底层向量库对 inner product 支持更好，我会先把 embedding 做归一化，再用 inner product 去实现，效果上接近 cosine，但工程上可能更方便；本质上我的选择还是“优先比较语义方向，而不是放大模长差异”。

另一个版本：RAG 里常见的相似度度量主要是 cosine、dot product 和 L2。我的理解是，cosine 更看语义方向，dot product 还会把向量模长算进去，L2 则是整体几何距离。如果 embedding 做了归一化，那 cosine 和 dot product 基本等价，L2 的排序也会比较接近；但如果不做归一化，dot 和 L2 会更容易受模长影响。结合我的科研资料知识库场景，我会更倾向于用 cosine，或者先归一化再用 inner product，因为我更关心 query 和 chunk 的语义匹配，而不是向量长度本身。对论文、教材、笔记这种长度和表达差异比较大的文本，这样通常会更稳。

**补充**：
在 RAG 的向量检索里，常见的相似度度量主要有 cosine similarity、dot product 和 L2 distance。它们本质区别在于：到底更关注方向、还是同时把向量长度也算进去。对 embedding 来说，这一点很关键。
**先说 cosine similarity**。它比较的是两个向量的夹角，本质上更关注语义方向是否接近，而不太受向量模长影响。所以如果 embedding 的模长本身没有特别明确的业务含义，或者不同文本长度、不同句式会带来向量长度波动，那 cosine 通常更稳。尤其是在很多通用文本检索场景里，大家更关心“语义是否相近”，而不是谁的向量更大，因此 cosine 是很常见的选择。
**再说 dot product**。它既看方向，也会把向量长度算进去。如果 embedding 模型训练时就默认用内积空间，或者向量模长本身携带了额外信息，比如模型用更大的模长表达更强置信度，那 dot product 可能更合适。但前提是你得清楚模长是不是有意义；如果模长只是训练和文本长度带来的副产物，那直接用 dot product 可能会把一些不该放大的因素放大。
**L2 distance 更偏几何距离**，衡量的是两个点在欧氏空间里的绝对远近。它在某些向量库和近邻搜索实现里也很常见，但它会同时受到方向和模长影响。如果 embedding 没做归一化，L2 对尺度比较敏感；如果做了归一化，其实它和 cosine 在排序上会比较接近，因为单位球面上欧氏距离和夹角是单调对应的。
这里 embedding 是否归一化 是个关键前提。如果向量做了 L2 normalization，那：cosine similarity 和 dot product 在排序上基本等价；L2 distance 也会和 cosine 呈单调关系。
也就是说，归一化以后，这三者差别会明显缩小，更多变成实现层和索引层的选择。但如果没归一化，那：cosine 更强调方向；dot 更强调方向加模长；L2 更强调整体几何距离；
这时候三者差异就会比较明显。

# 13、什么时候你会刻意不做 embedding 归一化，从而更倾向用 dot product（或 L2）来做召回？结合一个你熟悉的 RAG 场景，说清楚你想保留的“模长信息”通常代表什么信号。
我会刻意不做 embedding 归一化，通常是因为我认为向量模长本身携带了有用信号，比如匹配置信度、信息密度、内容质量，或者模型训练时学出来的表征强度。这种情况下如果归一化，等于把这部分信号抹掉了，所以我会更倾向用 dot product，让排序同时考虑方向和模长。结合我熟悉的 RAG 场景，比如科研资料知识库里混有教材定义段、论文正文和一些解析质量一般的 PDF 碎片时，我有时会希望模长能反映 chunk 的可靠性或语义集中度，让更完整、更高质量的内容排在前面。这个时候我想保留的模长信息，本质上不是文本长短，而是语义强度、质量或置信度信号。

# 14、你刚才说“embedding 模长可能反映 chunk 质量/置信度”，你会怎么验证这件事在你的知识库里确实成立？补充追问：如果你做完验证发现“模长和质量基本不相关”，你会用什么信号/机制来替代它，保证召回排序别被带偏？

如果我要验证“embedding 模长在我的知识库里是否真的反映 chunk 质量或置信度”，我不会只看直觉，而是会把它拆成一个相关性验证问题。核心思路是：先定义什么叫“高质量 chunk”，再看模长和这些质量指标是不是稳定相关。
第一步，我会先构造一批带质量标签的 chunk 样本。结合我的科研资料 RAG 场景，这些 chunk 可以按来源和可用性分几类，比如：教材或论文正文里结构完整的定义/方法段落、带清晰上下文的公式解释段落、普通背景描述段落，以及 PDF 抽取后质量较差的碎片文本、页眉页脚残留、表格错乱片段等。也就是说，我会先给一部分 chunk 做人工质量标注，至少区分成“高质量可直接支持问答”“中等质量相关但不完整”“低质量或噪声”这几档。这个项目本身就涉及 PDF 文本抽取、清洗、结构化分块、向量化存储和检索问答链路，所以这种按 chunk 可用性做分析是很自然的。
第二步，我会看模长和这些标签之间有没有统计上的相关性。最直接的方法有三类：
一类是看不同质量分组的模长分布，比如高质量 chunk 的模长均值、中位数是不是稳定高于低质量 chunk；
二类是算模长和人工质量分数之间的相关系数，比如 Spearman 相关；
三类是更贴近检索任务的离线实验：分别用“归一化后的 cosine 排序”和“保留模长的 dot product 排序”去跑同一批问答检索集，看哪种排序下的 Recall@K、MRR 或证据完整性更好。
因为对我来说，最终不是要证明“模长数学上相关”，而是要证明“它加进排序以后，召回结果是否真的更好”。
第三步，我会排除几个常见混淆因素，避免把“模长有用”误判成“文本长度有用”或者“噪声更大所以模长更大”。比如我会检查模长和 chunk token 长度的相关性，如果两者高度相关，那就说明模长可能只是长度代理变量，不一定代表质量；另外我也会按文档类型分桶，比如正文段、公式解释段、参考文献段、疑似噪声段分别看分布，避免整体上看起来有相关，但其实只是文档类型差异造成的。
如果最后验证下来发现：模长和质量基本不相关，那我就不会继续依赖它做召回排序，因为这说明不归一化可能只是在给排序引入噪声。这个时候我会改成“语义相似度做主排序，质量信号做显式建模”，而不是指望模长隐式表达质量。

具体替代方式，我会优先用这几类信号：
第一类是显式质量分。
比如在离线建库时，就给每个 chunk 打一个质量分，来源可以包括：是否来自正文主内容、是否命中标题层级、是否包含完整句群、是否被判定为目录/脚注/参考文献、OCR 或解析质量是否稳定、是否属于公式+解释的完整单元等。这样质量就不再依赖 embedding 模长，而是变成一个独立、可解释的特征。
第二类是两阶段排序。
先用归一化 embedding 做稳定的向量召回，保证语义相关性；再在 rerank 阶段把质量特征、文档来源、章节位置、chunk 完整性这些信号一起融合进去。也就是说，第一阶段解决“像不像”，第二阶段解决“好不好、值不值得进最终上下文”。
第三类是规则过滤 + 元数据约束。
比如把目录、参考文献、短碎片、页眉页脚残留这种低质量内容在入库前就过滤掉，或者在召回时降权；再利用我前面做的章节、小节、页码、是否公式解释段这些元数据，优先保留更完整、更像正文知识单元的 chunk。这样比把质量寄托在模长上更稳，也更容易解释。

# 15、你提到“引用返回、Prompt 约束”来提升回答准确性。你在离线评测里怎么定义并标注“引用一致性/证据可支撑性”——也就是区分“看起来相关的 chunk” vs “确实能支撑答案”的那条线？你会用什么判定规则让标注员可操作、可复现？

我会把“相关”和“可支撑”分开定义。相关只是主题接近，但可支撑要求这个 chunk 能作为回答核心结论的直接依据，也就是仅凭这段证据本身，或者它和同题允许组合的其他证据一起，就能推出答案里的关键陈述。标注时我会先把每个问题拆成几个核心答案点，再让标注员逐条判断候选 chunk：它是不是相关、能不能直接支撑某个答案点、如果能，支撑的是哪一个答案点。规则上我会明确禁止靠常识脑补，证据必须在标注范围内显式出现；背景信息默认不算支撑，除非问题本身就在问背景。这样就能把“看起来相关的 chunk”和“确实能支撑答案的 chunk”区分开来，后面评测时我也会单独看 Support@K，而不只看普通的 Recall@K。

# 16、如果一个答案点确实需要多个 chunk 组合才算“可支撑”，你在标注规范里会怎么定这个边界，才能做到可复现？补充追问：你会允许“跨 chunk 推理补全”（例如 A 给定义、B 给公式、需要你自己推一步才成立）吗，还是要求证据里必须把结论写出来？

如果一个答案点需要多个 chunk 组合才算可支撑，我会先给这个答案点定义“最小支持集合”和必需证据角色。只有当多个 chunk 合起来能把这些必需角色显式补齐，我才会把它标成 compositional support；如果还需要标注员自己基于常识或领域知识再往前推一步，我就不算它是可支撑。也就是说，我允许“证据拼接型补全”，比如 A 给公式、B 给变量解释，但不允许“自由推理型补全”，比如 A 和 B 相关，但必须靠自己推导才能得到答案。这样做的目的是把组合支撑的边界限定成可复现、可仲裁的规则，而不是凭感觉判断。

# 17、给你一个偏真实业务的场景：我们要做一个“研究助理”——用户用自然语言说“帮我把近三个月关于某个方向的论文做个对比表，并把关键信息写进飞书文档，同时把引用的 PDF 页码标出来”。系统可用的工具只有：search_papers、download_pdf、extract_citations、write_feishu_doc、create_table。

你会把这个需求建模成 Agent 还是主要靠 Function Calling 的单轮/多轮流程？选一个，并说清楚你这么选的关键原因（尤其是可控性、失败恢复、成本/时延）。
我会优先把这个需求建模成以 Function Calling 为核心的多轮编排流程，而不是一开始就做成强自治的 Agent。原因是这个任务虽然看起来长，但本质上是一个目标明确、工具集合固定、步骤相对可分解的流程型任务，更适合做成可控的“规划 + 分步调用 + 状态校验”的链路。

具体来说，这个需求其实可以被稳定拆成几步：先根据用户给的研究方向和近三个月这个时间范围调用 search_papers 找候选论文；然后对候选结果做筛选，按需要调用 download_pdf；接着从 PDF 里抽取关键信息和引用位置，调用 extract_citations 拿到可追溯的页码；之后把论文之间的维度整理成结构化字段，用 create_table 生成对比表；最后再通过 write_feishu_doc 把结论、表格和引用写进飞书文档。也就是说，它不是那种开放世界探索型任务，而更像一个有固定工具、有固定产物、有明确验收标准的工作流。

我之所以选 Function Calling 而不是更自由的 Agent，最关键是三个点。

第一是可控性更强。这个场景最终产出要写进飞书文档，还要求标明 PDF 页码，本质上是一个对准确性和可追溯性要求比较高的任务。如果用强自治 Agent，它虽然表面上更灵活，但也更容易出现工具乱调、步骤跳跃、没有真正完成引用校验就直接写文档的问题。相比之下，用 Function Calling 做成显式的多步流程，我可以在每一步都加校验点，比如论文是否真的在近三个月内、PDF 是否下载成功、引用页码是否抽取到、表格字段是否完整，最后再落文档。这样对业务会更稳。

第二是失败恢复更容易。这个任务天然会遇到很多局部失败，比如搜索结果不够准、某篇 PDF 下载失败、某篇论文 citation 抽取不完整、飞书写入失败。如果是 Agent 式黑盒推进，出错后往往很难定位它到底卡在哪一步；而 Function Calling 工作流可以天然保留中间状态，比如“已检索论文列表”“已成功下载的 PDF”“已完成 citation 抽取的论文集合”“待写入文档的结构化结果”。这样一来，某一步失败时可以只重试当前步骤，而不用整条链路推倒重来，恢复成本会低很多。

第三是成本和时延更可控。Agent 模式通常会引入更多自主思考和试错回合，工具调用次数也更不可预测；但这个业务其实不太需要那种开放式探索，更需要稳定地把任务做完。所以我会更倾向于让模型先做轻量规划，然后按固定模板逐步 Function Calling。这样可以控制 token 消耗，也能避免因为不必要的反复推理拉长时延。尤其在真实业务里，如果用户只是想要一份对比表和飞书文档，他更关心的是“能不能稳定产出”而不是系统表现得像不像一个很聪明的 Agent。

# 18、你刚才把它做成“以 Function Calling 为主的多轮编排 + 状态机”。那我们落到可执行层面：你来设计 extract_citations 这个工具的 function schema（参数、必填/可选、类型约束/枚举、返回结构），要求能支持“把引用的 PDF 页码标出来”这个验收标准。
我会把 extract_citations 设计成一个面向“信息点取证”的工具，而不是简单返回几个相关段落。因为这个场景最后要把关键信息写进飞书文档，还要标明 PDF 页码，所以它的输出必须天然支持“可追溯”。入参上我会至少设计 pdf_ids 和 mode 两个必填字段，mode 可以分成 claim_grounding、field_extraction 和 question_answering 三类；然后根据 mode 再传 claims、fields 或 question。另外我会加 max_citations_per_item、evidence_granularity 和 support_level 这些参数，去控制每条信息最多返回几条引用、返回页级还是段落级证据，以及区分“只是相关”还是“真正可支撑”。返回结构里，我会要求每条结果至少带 pdf_id、item_key、normalized_answer、support_status 和 citations。其中 citations 里最关键的是 page、snippet、support_level 和 confidence，必要时还能带 bbox 做页内定位。这样后面生成对比表时可以直接用标准化字段，写飞书文档时也能直接写成“这条结论来自某篇论文 PDF 第几页”。我觉得这个 schema 的核心不是字段堆得多，而是它能直接支撑最终验收：信息抽出来了，而且引用位置是清楚、可回溯的。

**详细版：**
如果让我设计 extract_citations 这个 function schema，我会把它设计成一个面向“信息点取证”的工具，而不是简单返回几个相关段落。因为这个业务最后的验收标准很明确：不仅要抽出论文里的关键信息，还要能标明这些信息来自哪篇 PDF 的第几页。所以这个工具的核心目标是：给定 PDF 和待抽取的信息点，返回结构化答案 + 对应页码 + 证据片段 + 支撑强度。
入参上，我会先要求两个必填字段：
第一个是 pdf_ids，类型是字符串数组，表示要处理哪几篇 PDF，这个肯定必填，因为后面所有引用都要绑定到具体文档；
第二个是 mode，我会把它设计成枚举值，比如 claim_grounding、field_extraction、question_answering。因为这个工具可能有三种典型用法：一种是给定一句待写入文档的话，让它去找证据；一种是直接抽结构化字段，比如方法、数据集、指标、结果；还有一种是用户直接问问题，让它返回带引用的答案。这样一个工具就能覆盖研究助理场景里“做对比表”和“写总结”的两类需求。

如果 mode 是 field_extraction，我会再要求一个 fields 字段，类型是字符串数组，里面是要抽的字段名，比如 method、dataset、metrics、results、limitations。
如果 mode 是 claim_grounding，那就传 claims，类型也是字符串数组，表示我要核验哪些陈述。
如果是 question_answering，就传一个 question 字段。也就是说，这三个字段不是都必填，而是根据 mode 互斥必填。

此外我还会加几个很关键的可选参数：
一个是 max_citations_per_item，控制每个字段或者每个 claim 最多返回几条引用，避免结果太散；
一个是 evidence_granularity，我会做成枚举，比如 span、paragraph、page，因为有时候下游只需要页码，有时候需要段落引用，有时候还要做高亮定位；
还有一个是 support_level，这个很重要，我会让它区分 relevant、supporting、strict_supporting。因为业务里不是“看起来相关”就够了，而是要区分这条证据到底只是相关，还是能真正支撑要写进飞书文档的结论。

返回结构上，我会让每个结果至少包含这几个核心字段：
第一个是 pdf_id 和 pdf_title，明确证据来自哪篇论文；
第二个是 item_key，表示当前这条结果对应的是哪个字段、哪个 claim 或哪个问题；
第三个是 normalized_answer，也就是标准化后的抽取结果，比如方法名、数据集名、结果数字，这个方便后面直接写表格；
第四个是 support_status，我会把它设计成枚举，比如 not_found、relevant_only、partially_supported、supported、strictly_supported，这个字段的作用是告诉下游：这条信息到底能不能直接写进飞书文档。

最关键的是 citations 字段，它一定要是个数组，因为一个信息点可能对应多处证据。每条 citation 里我会至少放四个核心信息：
一个是 page，这个是硬要求，因为题目里明确说了要把 PDF 页码标出来；
一个是 snippet，也就是页内证据片段，方便人工复核；
一个是 support_level，说明这条引用只是相关还是能严格支撑；
还有一个是 confidence，给后续系统做排序或人工审核参考。
如果工程上还要支持 PDF 高亮，我会再加一个可选的 bbox，也就是页内坐标框。

我之所以这么设计，是因为这个工具不是单纯给模型“看看材料”，而是要服务最终交付。也就是说，后面的 create_table 可以直接用 normalized_answer 去生成对比表，而 write_feishu_doc 可以直接把 snippet + page 写成“见某论文 PDF 第 5 页”这种形式。
同时 support_status 又能保证系统可控：如果只是 relevant_only，那就不能自动写入正式结论；如果是 supported 或 strictly_supported，才允许自动落到文档里。
所以如果我用一句话总结这个 schema 的设计原则，就是：输入要明确“抽什么”，输出要明确“抽到了什么、来自哪里、能不能直接作为引用写进文档”。

# 19、你刚才说 extract_citations 有 mode，并且希望做到“mode=claim_grounding 时 claims 必填、mode=field_extraction 时 fields 必填”这种条件必填。不用写很长，你直接口述/写出一段 JSON Schema 的关键结构就行：你会用什么机制表达这种约束（比如 oneOf/anyOf/allOf 或别的），以及在 claim_grounding 这个分支里你会如何限制 claims 里每个元素的字段类型与必填项？

# 21、你这个 extract_citations 已经能返回 page/snippet 了；现在假设线上经常出现一种问题：返回的页码确实来自这篇 PDF，但 snippet 跟 claim 对不上（看起来是“相关段落”，但不能支撑 claim）。你会把它优先归因到哪一类失败（解析失败 / 选错工具 / 参数错 / 执行错 / 结果对齐错 里选一个主因），以及为了定位它、改进它，你会补采集/标注一份什么样的最小数据集（样本长什么样、标什么字段）？
我会优先把这个问题归因到结果对齐错，而不是解析失败。因为页码已经对了，说明文档选择和页级定位大概率没出大问题，真正的问题更像是系统在页内摘错了段，或者把“主题相关”误判成了“可以支撑 claim”。为了定位这个问题，我会补一份最小的 page-level grounding 数据集，每条样本至少包含 pdf_id、page、claim_text、claim_type、这一页的候选 snippet、系统预测 snippet、gold supporting snippet，以及更细的标签，比如 direct_support、relevant_but_not_supporting、wrong_page_region、insufficient_snippet。必要时我还会补一个 missing_roles 字段，标明当前 snippet 缺的是 comparison、result 还是 variable explanation。这样我就能区分到底是页内切分太粗、supporting 判别不够严格，还是某些 claim 类型天然容易错。后续优化上，我会先看要不要把页内粒度切细，再考虑补一个专门判断 supporting 的 reranker，而不是只用相关性排序。
我的重点不是先做一个很大的数据集，而是先做一份能直接解释线上失败原因的最小集。因为只有先把“错在哪里”分清楚，后面改模型、改切分还是改规则，才不会盲目。

# 22、你先把 SFT 和 RLHF / DPO 这类偏好对齐的区别讲清楚：它们各自的训练目标是什么、最典型的数据长什么样（样本结构/字段），以及你觉得各自最容易踩的数据坑是什么？
我理解 SFT 和 RLHF / DPO 的核心区别是，SFT 更像教模型“这类问题应该怎么答”，它的训练目标是监督学习，最大化目标答案的条件概率，所以最典型的数据就是 prompt 加 response 这种单答案结构。它比较适合做基础能力塑形，但容易踩的坑是标注风格不一致、一个问题只保留一种写法导致模型变僵，以及低质量答案被模型直接学进去。
RLHF 和 DPO 则更偏“偏好对齐”，解决的是多个答案都还行时，模型应该更偏向哪一种。RLHF 通常是先用 prompt、response_a、response_b、preference_label 这类比较数据训练 reward model，再用强化学习优化；DPO 则更直接，用 prompt、chosen、rejected 这种偏好对直接做训练。它们最容易踩的坑是偏好标准不一致、chosen/rejected 差异不干净，以及模型学到一些表面代理特征，比如更长、更保守就一定更好。我的理解是，SFT 更难在覆盖和质量平衡，偏好对齐更难在“什么叫更好”这件事的数据定义。

# 23、你做 SFT 数据时，去重你会怎么做：你会定义哪些“重复”的判定口径（比如完全重复/近重复/语义重复），以及各自用什么方法落地（规则、hash、embedding 聚类等）？
如果我做 SFT 数据去重，我一般不会只做一种“完全重复删除”，而是会把重复分成至少三层来看：完全重复、近重复、语义重复。因为这三类重复对训练的影响不一样，处理方式也不一样。我的目标通常不是把所有相似样本都删光，而是尽量避免模型被同一种样本过度刷屏，最后学到偏置分布。

先说 完全重复。这个最直接，就是输入和输出都一样，或者标准化之后完全一样。比如同一个 prompt-response 样本因为数据汇总、版本回流、多人标注合并，被重复收进来了三次。这类重复我会优先用规则和 hash 做。具体上会先做一轮文本标准化，比如去掉多余空格、统一换行、全半角统一、JSON key 排序、时间戳或 trace id 这类明显无关字段清洗，然后对 prompt + response 做 hash。这样可以快速去掉最机械的重复。如果是多轮对话数据，我会对“system + history + user + assistant”拼接后再 hash，而不是只看最后一轮。因为很多看起来最后一问一样的样本，历史上下文其实不一样，不能误删。

再说 近重复。这个比完全重复更常见，比如 prompt 基本一样，只是改了个标点、换了个同义表达，或者 response 只有措辞轻微变化，但任务本质完全一样。比如“请总结下面文章”与“帮我总结以下文章”，或者工具调用样本里只是 request id 不同、字段顺序不同，但调用逻辑完全一致。这类我一般会先走规则层，再走字符串相似度。规则层常见做法包括：对模板化字段做归一，比如 URL、数字编号、日期、文件名、随机 id 先替换成占位符；对 JSON、函数调用参数做规范化序列化；对代码或 SQL 做格式化。做完这些以后，再用编辑距离、Jaccard、n-gram overlap、SimHash/MinHash 这类轻量方法筛一遍。我的经验是近重复很多时候不是“语义理解问题”，而是“模板噪声问题”，先用规则清洗能解决一大半。

第三类是 语义重复。这个更 tricky。比如两个 prompt 表面文本不一样，但问的是同一个任务，response 也只是换一种说法，本质上是在重复教模型同一件事。举个例子，“解释一下 Transformer 的 self-attention” 和 “什么是自注意力机制，它在 Transformer 里起什么作用”，如果回答内容也高度重合，那这两条对模型来说可能就是高语义重复。再比如工具调用场景里，“帮我查近三个月某方向论文并整理表格”和“请检索最近三个月的该领域论文，做成对比表”本质流程几乎一样。这类我不会只靠规则，而是会结合 embedding 做召回或聚类。常见做法是：先把样本拆成适合表达任务语义的字段，比如 instruction、task summary、response type、tool trace 摘要，做 embedding；然后用 ANN 检索相似邻居，或者做局部聚类。真正落地时，我一般不会直接“聚完就删”，而是先把高相似簇打出来，再结合一些结构特征决定保留哪条。因为语义相似不等于一定重复，有时只是同一技能下的不同边界条件，删得太狠会损失覆盖。

# 24、你刚才说“语义去重聚簇后不直接删”，那你在一个高相似簇里通常按什么原则决定保留哪些样本、删除哪些样本？补充追问：如果簇里样本质量参差（有的回复更好但更长/更保守），你怎么避免把“长度/口吻”这种表面特征当成保留标准？

我在高相似簇里一般不会直接“留一删其余”，而是先判断这些样本是不是在教同一件事。如果只是表面相似，但约束条件、异常路径、输出目标不一样，我通常都会保留；只有当监督信号真的高度同质时，才做簇内筛选。筛选时我主要看四类东西：任务约束是不是更完整，输出格式是不是更规范，是否覆盖了更有价值的边界情况，以及有没有提供新增监督信息。删除的通常是模板重复、低质量或者对已有样本没有新增价值的样本。至于簇里样本质量参差时，我不会把“更长”“更保守”“更礼貌”直接当成更优，而是先按任务完成度、事实性、格式合规性、约束遵守度这些任务相关维度来判断，再去看长度和风格是不是在真正服务这些目标。这样能避免把长度、口吻这种表面特征误当成保留标准。

# 25、你刚才提到簇内筛选会看“有没有新增监督信息”。你怎么把“新增监督信息”落到一个可执行、可复现的判定流程上：给定一个高相似簇，你会用哪些可量化的信号/规则来判断某条样本是“新增”还是“重复刷屏”？（尽量说到你会怎么自动化跑、以及最后人工只需要看什么）

我会把“新增监督信息”定义成：这条样本相对于簇里已保留样本，是否新增了任务约束、执行路径、输出结构或者错误恢复模式中的至少一项。落地上我不会直接看原始文本，而是先把每条样本抽成结构化特征，比如任务类型、约束标签、工具调用路径、输出槽位和异常处理，再生成一个监督签名。自动化筛选时，我会先按质量分排序，再做簇内增量保留：如果一条样本和已保留样本在这些关键维度上都没有新增，只是措辞、长度或口吻不同，我就把它判成重复刷屏；如果它新增了时间范围、引用要求、失败恢复路径、参数补全逻辑，或者新的输出字段，我就保留。最后人工只需要看边界样本和高质量冲突样本，而且看的是压缩后的结构化差异，而不是从头读全文。这样“新增监督信息”就不是一个主观感觉，而是可以自动跑、也方便复核的一套 coverage 增量判断流程。

# 26、你刚才提到“先按质量分排序，再做簇内增量保留”。这个质量分你会怎么定义成可量化的打分：你会用哪些字段/信号来算分、各自大概怎么加权？补充追问：你怎么验证这个质量分不会系统性偏向“更长/更安全措辞”的答案？

我会把这个质量分定义成一个面向任务完成度的综合分，而不是“读起来更像好回答”的印象分。因为在 SFT 去重里，这个分数的作用不是选“文风最好”的样本，而是选更适合作为监督信号的样本。所以我一般会把它拆成几类可量化维度，每类都尽量和任务目标直接相关。
具体上，我通常会看五类信号。
第一类是任务完成度，这会是权重最高的一类。核心看这条样本有没有把用户要求的关键约束做全，比如该回答的问题有没有答到、该提取的字段有没有全、该调用的工具有没有走完、该给的引用有没有给。这类我一般会给最高权重，差不多会占 35% 到 40%。因为一条样本如果任务本身没完成，其他再好也不适合拿来做 SFT 监督。
第二类是事实性 / 一致性。比如回答里的内容是否和上下文一致、有没有明显幻觉、工具调用结果和最终回答是否一致、引用和结论能不能对上。这类我一般会给 25% 到 30% 的权重。因为 SFT 最大的风险之一就是把错误答案直接教给模型，所以事实一致性一定是高权重项。
第三类是格式与结构合规性。这类主要看输出是不是符合任务预期格式，比如该输出 JSON 的有没有合法 JSON、该做表格的有没有结构化字段、该做 function call 的参数是否齐全、字段命名是否规范、是否有明显脏格式。这类我通常给 15% 左右。因为格式问题虽然不一定影响语义，但在工具调用、结构化输出、Agent 场景里影响很大。
第四类是约束遵守度。比如用户要求“只基于给定材料”，它有没有越界发挥；要求“简洁回答”，它有没有过度扩写；要求“标明页码”，它有没有把页码写出来；要求“近三个月”，它有没有跑到更早的内容里去。这类我一般给 10% 到 15%。它和任务完成度有点像，但更强调有没有真正遵守边界条件。
第五类是信息密度 / 冗余控制。这类我会比较谨慎地用，通常只给 5% 到 10%，而且不会直接把“更长”记高分。我真正想衡量的是：在完成任务的前提下，这条样本是不是有大量重复表达、空洞套话、过多免责声明。也就是说，我不是奖励长，而是在轻微惩罚“无效扩写”。

# 27、你简历里 PM2.5 项目写了“7 天预测任务上，相比 TimesNet，北京 MAE -12.1%、RMSE -7.2%”。你把这个提升的可信度压实一下：你当时北京数据集的训练/验证/测试是怎么切的（按时间切还是随机切），TimesNet 的对比是用同一份划分、同一套特征与相同预测设置跑出来的吗？（如果有调参，调参只用到了哪一部分数据）

这个结果我当时是按比较标准的时序预测实验口径去做的，数据划分是按时间切的，不是随机切分。因为这是 PM2.5 的长期时序预测任务，如果随机切分，会把未来信息泄漏到训练集里，结果会偏乐观，所以我当时更强调按时间顺序去划 train、validation 和 test，比例为7:2:1。我的理解是，训练集只用历史一段时间，验证集用紧接着训练集之后的一段时间做调参和 early stopping，测试集再用最后一段完全未见过的连续时间窗口做 7 天预测评估，这样更符合真实 forecasting 场景。

跟 TimesNet 的对比，我会坚持同一份数据划分、同一套输入特征、相同预测 horizon、相同评估指标来跑，不会给我的模型和 baseline 用不同口径。也就是说，北京数据集上报出来的 MAE 和 RMSE 提升，前提应该是两边都在同样的 train/val/test 划分、同样的 7 天预测设置下得到的；不然这个提升没有说服力。简历里写的北京数据集 MAE 降低 12.1%、RMSE 降低 7.2%，我在理解上就是建立在这种同口径对比上的。

调参这块，我会明确说明：调参只用训练集和验证集，不碰测试集。比如模型结构、学习率、batch size、图构造相关超参数、early stopping 这些，都应该是在 train 和 val 上完成选择，测试集只在最后做一次报告结果。因为如果测试集参与了调参，哪怕没有直接训练，也会让最终提升的可信度打折。

# 28、你这组 “北京 MAE -12.1%、RMSE -7.2%” 是单次跑出来的最好结果，还是多 seed / 多次重复实验后的均值？补充追问：如果是多次重复，你最后在简历里写的提升是怎么做汇总的（均值/中位数/最好/加权），以及方差大概在什么量级？

我这个结果目前是单次实验结果。如果现在要用最小成本把它补成一个可复现、也更有统计意义的结论，我会固定同一份按时间顺序切的 train/val/test 划分，不改测试集；然后让我的模型和 TimesNet 在完全相同的特征、预测 horizon、评估指标和 early stopping 口径下，各跑 3 到 5 个随机种子。每次只改变初始化和训练随机性，验证集负责选模，测试集只在最后评一次。最终我会报告均值和标准差，而不是继续写单次最好结果。比如简历里我会更愿意写成“在北京数据集 7 天预测任务上，基于固定时间划分和 5 个 seed 重复实验，相比 TimesNet，MAE 平均下降 xx% ± xx%，RMSE 平均下降 xx% ± xx%”。如果还没补完多 seed，我也会主动把表述改成“一次代表性实验中取得的结果”，这样我自己才敢背书。

# 29、如果你按你说的补了 5 个 seed，结果发现：MAE 平均只下降 1% 左右、标准差很大，甚至有 2 个 seed 还不如 TimesNet——你会怎么判断这到底是“改进不稳/不显著”，还是“实验设置里有问题”？你会优先补哪一类验证来定位原因？
如果补完 5 个 seed 以后发现平均 MAE 只下降 1% 左右、方差又很大，甚至有 2 个 seed 还不如 TimesNet，我第一反应不会再说模型稳定优于 baseline，而是先按更保守的口径理解成：目前这个改进不稳，至少还谈不上显著。接下来我会先区分到底是模型本身不稳，还是实验设置放大了波动。优先做的验证有三类：第一类是训练稳定性验证，看不同 seed 的 train/val 曲线、early stopping 选中的 checkpoint 和收敛情况是不是差异很大；第二类是关键超参数敏感性验证，看学习率、窗口长度或者图结构相关参数是不是导致结果特别飘；第三类是在保持时间顺序前提下做相邻时间切点验证，看结论是不是对测试窗口特别敏感。如果这些检查之后结果还是小幅提升、方差大、不同 seed 交叉领先，我会更坦率地判断这个改进目前不稳或不显著，而不是继续坚持它优于 TimesNet。这样我觉得结论才是自己敢背书的。
