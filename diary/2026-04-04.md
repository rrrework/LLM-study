# 基于目前自己的RAG科研资料问答助手的一些思考
目前这个 RAG 系统 完全没有使用 LangChain（或其他类似开源框架），而是纯手动（raw / from scratch） 搭建的。
1. 目前是怎么搭建的？
我给你发的所有代码（包括 PDF 读取 + chunk 切分 + Chroma + OpenAI 调用）都是直接使用底层库：

PDF 读取：用 pymupdf (fitz) 直接打开 PDF 提取文本。
文本切分：用 langchain_text_splitters 中的 RecursiveCharacterTextSplitter（只借用了它的 splitter 工具类，并没有用整个 LangChain 框架）。
向量存储与检索：直接用 chromadb 的 PersistentClient 和 collection.add() / collection.query()。
Embedding 和生成：直接调用 openai 官方 SDK（client.embeddings.create 和 client.chat.completions.create）。
整体流程：自己手动写函数把这些步骤串起来（读取 → 切分 → embedding → 存入 Chroma → 查询 → 构建 messages → 流式调用 LLM）。

这种方式也叫 vanilla / raw Python + direct SDK 实现。代码完全透明，每一步你都能看到到底在做什么，没有任何“黑盒”抽象。
2. 为什么没有用 LangChain？
主要原因有以下几点（非常适合你当前的科研资料问答助手场景）：

简单清晰，易于理解和调试
你是想深入学习 RAG 的工作原理（embedding、chunk、检索、prompt 构建等），用 raw 方式能把每一步都看得很清楚。LangChain 会把很多步骤封装成 DocumentLoader、Retriever、Chain 等抽象，初学者反而容易“看不见”底层发生了什么。
轻量、依赖少、性能更好
LangChain 是一个很大的框架，引入后会带来较多依赖包、版本冲突风险，以及一定的运行开销（尤其在简单场景下）。我们目前的代码只用了必要的几个库，运行更快、更稳定。
更灵活、可控
你可以随意调整 chunk 大小、overlap、相似度计算、prompt 模板、重试逻辑、多模型切换等，而不用受框架约定的限制。科研场景经常需要精确控制 prompt（低 temperature、严格基于论文回答），raw 方式改起来更直接。
适合当前规模
你的项目目前是科研论文问答，核心是“读取 PDF → 切 chunk → 存向量 → 检索 → 生成”，属于经典的 Naive RAG。这种简单流程用 raw 代码写几百行就够了，不需要框架带来的复杂性。

很多开发者在 2025-2026 年也反馈：对于简单到中等的 RAG 系统，纯手动实现往往比用 LangChain 更清晰、更好维护，尤其当你想深入理解或做自定义优化时。
3. 什么时候才会考虑引入 LangChain（或 LlamaIndex）？

项目变复杂了：需要多轮对话记忆、Agent（智能体）、工具调用（比如让 AI 同时搜索网页 + 查论文）、路由（不同问题走不同检索策略）、评估（Evaluation） 等高级功能。
需要快速集成大量数据源（上百种文档类型、数据库、API）。
团队协作开发，想用标准化的组件快速搭建原型。
想用 LangSmith 等 observability 工具做 tracing 和调试。

如果只是“把论文放进去，能基于内容准确回答问题”，目前这种 raw + Chroma 的方式已经非常足够，而且更适合学习。
总结一句话：
我们目前是用最直接、最透明的方式搭建的 RAG 系统，没有用 LangChain，正是为了让你清楚地看到 RAG 的每一个核心环节（PDF 处理、切分、向量存取、检索、prompt 构建、调用）。等以后你想加复杂功能（如多 Agent、自动评估、混合搜索）时，再考虑引入 LangChain 会更合适。
