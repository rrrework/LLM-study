**Skill是封装了Tools的高级体**
你可以这样理解：
用户问题
 -> Skill Router 判断意图
   -> 如果是“怎么接”
      -> 调文档检索 tool
      -> 调代码检索 tool
      -> 生成接入步骤
   -> 如果是“报错了”
      -> 调错误解析 tool
      -> 调排障规则 tool
      -> 生成修复建议
   -> 如果是“上线前检查”
      -> 调配置检查 tool
      -> 输出风险清单
这里真正“干活”的是 tool；但负责把它们串起来、决定顺序、维持角色一致性的，就是 Skill。
和普通 function calling 的差别：
如果你只做 function calling，常见形态是：
模型知道有 search_docs()
模型知道有 search_code()
用户一问，模型自己决定调哪个，这只能算“有工具能力”，而当你继续往上做这些东西时，才更接近 Skill：
- 固定领域边界
- 固定决策流程
- 固定知识源优先级
- 固定输出模板
- 固定质量约束
- 固定异常处理
**也就是说，Skill 不是替代 tool，而是把 tool 产品化。**
