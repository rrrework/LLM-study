1. 介绍RAG项目，整体架构和实现思路

基于 RAG 的科研资料问答助手

我自己做过一个基于 RAG 的科研资料问答助手，主要是为了解决自己在做研究时查论文、看教材特别低效的问题。平时会积累很多 PDF 论文、教材章节和读书笔记，分散在本地和网盘里，传统搜索只能按标题或关键词，很难快速定位到某个概念在具体论文里的表述和推导过程。

整体上我把系统分成两个部分：离线构建知识库和在线问答。离线阶段，我先从论文 PDF 和笔记中抽取文本，接着对文本进行清洗（去除一些页眉页脚、重复的标题等），接着按论文结构做chunk，比如按章节、小节、带公式的段落来切分，每块大概 200–400 个 token，尽量保证每个块对应一个比较完整的小知识点；同时为每个块打上元数据，比如论文标题、作者、年份、会议、章节名、是否包含公式等。然后使用一个句向量模型为这些 chunk 生成 embedding，存入本地的向量库（Chroma），同时保存原文和元数据，方便后面做语义检索和过滤。

在线问答阶段，当我提一个问题，比如“Transformer 里 multi-head attention 的公式推导是什么”或者“这篇论文的 loss 是怎么设计的”，系统会先对这个问题做 embedding，在向量库里通过做计算余弦、内积等操作得到相似度，再对其做top-k和re-ranking操作，从而能结合论文来源和章节信息检索出若干最相关的段落；有些问题我还会限制只检索某几篇指定论文或某一门课程的讲义。然后把这些检索到的片段作为上下文，与用户问题一起构造成 prompt，交给大模型生成回答，并且在回答里附上引用的论文标题和页码位置，方便我回到原文核对。

实现过程中我重点做了两方面优化：一方面是 chunk 和检索策略，一开始我用固定长度切分，结果经常把公式和解释割裂开，检索回来上下文不完整，后来改成按论文结构切 + 长度上限控制，相关性提升明显；同时对检索结果做了二次重排序，让真正与问题最相关的几段出现在前面。另一方面是通过 prompt 约束模型的行为，明确要求它“只根据给定文档内容回答”，如果检索结果很少就直接说“不确定”，并且尽量给出原文引用，来降低幻觉风险。

实际使用下来，这个小助手在做文献回顾、对比不同论文对同一概念的定义时帮助很大，很多原来需要来回翻 PDF 才能确认的细节，现在只要问一两句就能在自己的文档集合里快速定位和对比，大幅提升了我自己查文献和写相关工作部分的效率。

1. 离线构建知识库在干嘛
离线构建知识库，可以理解成“给自己的文档做一个可检索的索引”，通常包括几步：
（1）收集数据：你的论文 PDF、教材章节、笔记等。
（2）清洗 + 切分（chunk）：按章节、小节、自然段，把长文档切成小块，每块语义尽量完整。
（3）Embedding：用一个 embedding 模型 把每个 chunk 变成向量（高维数字）。
（3）存储：把“向量 + 原文 + 元数据（标题、章节、年份…）”存到你自己的向量库里（Chroma）。
这一步就是你说的“做自己的向量库”，是 你应用自己的“记忆”，和大模型本身是分开的。

2. 在线问答用的是哪个向量库
在线问答阶段流程是这样的：
（1）用户提问。
（2）把这个问题也用同一个 embedding 模型转成向量。
（3）拿“问题向量”去 你自己的向量库 里做相似度检索，找出最相关的若干个 chunk。
（4）把这些 chunk 的原文拼成上下文，和用户问题一起放进 prompt 里，发给大模型，让它生成答案。
这里有两个关键点要记住，用来回答你刚才的疑惑：
大模型“内部确实有它预训练时学到的知识”，但它 没有一个你可以直接用的“内部向量库接口”，RAG 的检索完全是你自己外部做的。RAG 场景下，检索只用你自己构建的向量库（你自己的论文/笔记）。大模型只是一个“读你给的上下文 + 生成回答”的组件，它不会再去查一个“自己的向量库”。

如果面试官问你：“那大模型自己的知识在哪里用上了？”你可以这样说：
模型内部的知识用在：理解问题、理解检索到的片段、帮你整合和改写答案。
你自己的向量库用在：保证回答一定基于你自己的科研资料，而不是模型自己胡乱回忆或编造。

# 问题1：在这个科研资料问答助手里，大模型部分我没有自己从零训练，而是通过 API/本地部署调用现有的大模型。
我的主要工作是搭建 RAG 流程：把论文和笔记向量化存入向量库，查询时先做语义检索，再把检索到的片段和用户问题一起构造成 prompt 交给大模型生成答案。大模型在这个系统里更像一个插件，我也预留了接口，后面可以很方便地切换成本地开源模型。
# 问题2：问“你一般怎么配这些参数”“为什么这么配”。你可以提：
模型选择：根据场景选不同大小的模型，比如科研助手用中大型模型保证推理和代码理解能力，如果成本敏感会提供小模型作为备选。
温度（temperature）：问答 / 文档助手用低温度（0–0.3），保证稳定、少胡编；创意写作可以调高一点。
最大输出长度（max_tokens）：控制成本和响应时间，在 RAG 里根据上下文长度和需要的回答详细程度设一个上限。
Top_p / Top_k：有需要再调，一般和 temperature 配合控制多样性。
# 在科研问答助手里，我通常把温度设得比较低，让回答更确定；max_tokens 根据回答期望长度做限制，避免模型跑太长影响延迟和成本。

在这个科研资料问答助手里，大模型部分我是通过标准的 HTTP API 调用的，既支持同步整段返回，也支持流式输出，前端可以实时展示回答。调用时我一般使用 chat-completion 风格的接口，把系统角色、检索到的论文片段和用户问题一起作为 messages 传给模型。参数上会把 temperature 设得比较低，保证回答稳定可靠，同时控制 max_tokens 防止输出过长。工程上还做了并发和重试控制，比如对超时和 429 做退避重试，必要时可以在不同模型提供方或者本地模型之间切换，以保证整体可用性。

# 问题3：检索错召（召回结果不相关）怎么处理？
如果发现检索结果相关性不高，我会先在 dense 检索后加一层 rerank，用更强的交叉编码器对 top-K 结果重新打分，只保留最相关的几条作为上下文。同时结合元数据做过滤，比如限定领域、时间或文档类型，减少场景完全不匹配的噪音；必要时回头调 chunk 策略，避免一个 chunk 里混多个主题导致“看起来相关其实跑偏”。

# 问题4：检索漏召（该召回的没召回）怎么处理？
漏召本质是 recall 不足，我会先通过增大 top-K、引入 hybrid search（语义向量 + 关键词/BM25）来扩充候选集合，再用 rerank 保证前几条结果质量。对复杂 query 会做 query 重写或扩展，比如加入同义词、常用缩写，必要时更换或微调更适合领域文本的 embedding 模型，并调索引参数来提高 recall。

# 问题5：RAG 效果怎么评估？用哪些指标？
我把评估拆成两块：检索层面看 context precision / recall，简单说就是“取回的上下文有多相关”和“理想需要的信息有多少被取回来”。生成层面看 answer relevancy 和 faithfulness，前者关注“回答是不是在答这个问题”，后者看“回答是否完全被检索到的上下文支撑”，faithfulness 低就说明有幻觉；再结合一些端到端指标，比如任务成功率和用户主观评分。

# 问题6:chunk 策略怎么优化？（被追问 chunk 时用）
chunk 我不会只按固定长度硬切，而是“结构优先”：先按章节/小节/标题边界切，再用最大长度控制，保证每个 chunk 对应一个尽量完整的小知识点。实践中我会对相邻 chunk 做少量 overlap，避免定义和解释被拆开；如果发现检索结果经常“主题混杂”或“信息不全”，就通过缩小 chunk 粒度或调整 overlap 做 A/B 测试，看对检索 precision/recall 的提升情况再定参数。


相关函数：
1、np.array():将列表、元组或其他序列转换为 NumPy ndarray（多维数组）；对应把 OpenAI/Sentence-Transformer 返回的 embedding 列表（Python list）转为数组，便于后续矩阵运算。
2、np.reshape():在不改变数据的情况下，给数组一个新的形状；对应rag系统中将一维查询向量转换到二维向量矩阵，对其进行形状匹配，以便做矩阵乘法计算相似度。
3、np.dot():点积操作，；对应rag系统中计算查询向量和文档向量的余弦相似度
4、np.linalg.norm():计算向量或距离的范数，默认是 L2 范数，即欧几里得距离的平方根；对应rag系统中余弦相似度公式中需要除以两个向量的范数。
5、np.argsort():返回数组排序后的索引；对应rag系统中的top-k排序
