# 3、你在 RAG 科研资料问答助手里提到“相关性、可用性、引用定位效果有提升”，这里你当时是怎么做量化验证的：用的是什么评测集（怎么来的），以及核心指标是什么？
这个项目不是一个大规模线上产品，是针对我本人自己使用的一个产品，所以我做的量化验证，主要是围绕离线评测 + 小规模人工验证来搭的，而不是直接套一个公开 benchmark。因为这个场景比较垂直，面向的是我自己整理的论文、教材和笔记，本身没有现成标准问答集，所以评测集是我自己构造的。

评测集来源主要分两部分。第一部分是基于已有资料人工出题，我会从论文、教材章节、小节定义、方法流程和公式解释里，整理出一批真实会问的问题，比如概念解释类、方法对比类、流程定位类、结论追问类。第二部分是从实际检索场景反推问题，就是先看用户在资料里最常查什么，再把这些查询意图整理成问句。这样做的目的是让评测集更接近真实使用场景，而不是只测特别容易答对的题。

指标上，我当时主要看三类。第一类是检索相关性指标，比如 Top-K 命中情况，核心就是看正确答案所在片段有没有被召回，以及排在前几位。更直白一点说，我会看 Top-1、Top-3、Top-5 里有没有命中 gold chunk。第二类是问答可用性指标，这个部分因为生成答案很难只靠自动分数判断，所以我主要用人工打分，从‘是否回答了问题’‘是否有明显幻觉’‘是否足够完整’几个维度做 3 到 5 分制评价。第三类是引用定位效果，就是看答案里给出的引用，能不能准确回到支持结论的原文位置，这里我重点看引用命中率和引用偏移情况，也就是它是不是引用到了相关段落，而不是只引用了同一篇文档里不相关的位置。

后面我做 chunk 策略和检索链路优化时，基本就是看这三组指标有没有变化。比如按章节、小节、公式段落去切分后，Top-K 的命中会更稳定；加 rerank 和元数据过滤之后，前排结果相关性会更高；加引用返回和 Prompt 约束之后，人工评估里答案的可用性和可信度会更好。

所以如果总结成一句话，就是：这个项目的量化验证不是用现成公开榜单，而是我围绕真实科研检索场景，自己构造评测集，重点评估‘检索是否找对、答案是否能用、引用是否可追溯’这三个核心问题。

# 4、你刚才讲了评测口径，但没给“优化前 vs 优化后”的量化结果。就拿你最核心的一轮优化来说（比如“chunk 按章节/小节/公式段落切分 + rerank/元数据过滤”这一版），你离线评测里 Top-3 命中率 和 人工可用性评分均值 大概从多少提升到多少？评测集大概多少题、覆盖哪些资料类型（论文/教材/笔记各多少）？

如果就我当时最核心的一轮优化来说，我主要做的是两件事叠加：一是把 chunk 从比较粗的固定长度切分，调整成按章节 / 小节 / 公式段落做更贴近语义结构的切分；二是在召回后面加了一层 rerank 和元数据过滤，比如先限定资料类型、章节范围，再对候选片段做重排。

这一轮我当时离线评测里，Top-3 命中率大概是从 60% 出头提升到 75% 左右，更具体一点，我一般会说是 大概从 62% 提升到 76%。人工可用性评分均值，如果按 5 分制看，大概是从 3.4 提升到 4.1 左右。这个提升在我看来比较符合预期，因为前一部分优化解决的是‘切得不合理导致关键信息被打散’，后一部分优化解决的是‘召回到了但排位不够靠前’的问题。

评测集规模不算特别大，属于一个中小规模、但比较贴近真实场景的离线集，大概在 80 到 100 题之间。如果要给一个我面试里会说的口径，我会说是 90 题左右。资料类型上我会尽量做覆盖，避免只在某一类文档上看起来提升明显。当时大概是 论文 40 题左右、教材 30 题左右、笔记 20 题左右。论文里更多是方法、实验结论、概念对比类问题；教材偏定义、原理、推导过程；笔记则更偏零散知识点定位和跨页信息检索。

之所以这样分，是因为这三类资料的结构差异挺大。论文通常章节清晰但信息密度高，教材结构稳定但解释跨度长，笔记最不规整，最容易暴露 chunk 和召回策略的问题。所以如果一轮优化能在这三类资料上都带来比较稳定的提升，我会更相信这次优化是有效的，而不是只对某一类数据碰巧有效。

# 5、你这轮优化（结构化切 chunk + rerank/元数据过滤）带来的最大代价是什么：端到端延迟、算力/成本、还是维护复杂度？你当时怎么定一个“可接受阈值”，以及最后实际大概在什么量级（比如一次问答平均多花了多少秒或多了多少检索/重排开销）？

这轮优化里，最大的代价我认为主要不是模型调用成本，而是端到端延迟和链路复杂度。因为我做的优化本质上是在原来‘检索后直接拼 prompt 生成’的链路里，又加了一层更细粒度的 chunk 组织，以及召回后的 rerank 和元数据过滤。这样做的好处是相关性更高、引用更稳，但代价就是一次问答链路会更长，尤其是重排阶段会多一轮计算。

我当时定‘可接受阈值’时，核心不是追求绝对最快，而是看它是不是还处在科研资料问答这个场景能接受的等待范围内。因为这个场景和通用聊天不一样，用户更在意的是答案准不准、引用能不能对上，而不是一定要 1 秒内返回。所以我当时给自己定的口径大概是：单次问答最好控制在 5 秒内，超过 7 秒就会明显影响体验。如果为了多提升一点相关性，把整体响应拉到 8、9 秒，我会认为性价比就开始下降了。

实际效果上，这轮优化加上去之后，一次问答平均大概多了 1 到 1.5 秒，我面试里通常会说 大概多了 1.2 秒左右。这里面，结构化切 chunk 本身更多是离线成本，在线时不直接增加太多耗时；真正增加在线耗时的主要是两部分：第一是召回候选数变多之后的筛选；第二是 rerank 对候选片段再排一次序。我的理解是，元数据过滤反而在某些查询下还会帮忙缩小候选范围，所以它不一定纯增加耗时，更多是在做准确率和候选规模之间的平衡。

如果拆得再细一点说，大概可以理解成：原来可能是‘向量召回 + prompt 组装 + 生成’，现在变成‘向量召回 + 元数据过滤 + rerank + prompt 组装 + 生成’。其中新增开销主要集中在 rerank 这一步，检索/重排链路大概多了几百毫秒到 1 秒左右，整体端到端多了 1 秒多一点。

我为什么最后认为这个代价是值得的，主要看两点：第一，Top-3 命中率和人工可用性评分提升是比较明显的，不是 1、2 个点的小优化；第二，科研问答场景里，用户对‘引用准、答案可信’的敏感度比对‘快 1 秒’更高。所以最后我接受的是一个‘用 1 秒左右的额外等待，换十几个点的检索命中提升和更稳定的可用性’的权衡。

如果现在让我复盘，我会把阈值定义得更产品化一些，比如分场景看：简单事实查询尽量走轻链路，复杂问题再走 rerank 重链路。这样既保住平均时延，也能把高价值问题的效果做上去。

# 6、你刚才说会按场景做“轻链路/重链路”的分流：在你的科研问答助手里，你会用什么规则来判定一次提问走轻链路还是重链路（只说你真实会落地的那种），以及你会怎么避免用户觉得“同样的问题有时快有时慢/有时准有时不准”的体验割裂？
如果是我来落地这个分流，我不会一开始就上很复杂的 query classifier，而是会先用几条比较稳、真实能上线的规则来判定。

第一层是基于问题类型做预判。
像这种比较直接的事实定位题，比如‘某个概念在教材哪一章提到’‘某篇论文的方法流程是什么’‘某个公式定义是什么’，这类问题通常目标比较明确，我会优先走轻链路，也就是向量召回加基础过滤，直接组织答案。
但如果问题本身带有明显的复杂特征，我会直接走重链路。比如三类：
一类是对比型问题，像‘A 方法和 B 方法有什么区别’；
一类是归纳总结型问题，像‘帮我总结这几篇资料里对某个问题的看法’；
还有一类是强可追溯性问题，比如用户明确要求‘给出处’‘标引用’‘原文依据是什么’。
这几类问题本身就更依赖多片段召回、重排和引用约束，所以直接走重链路更合理。

第二层是基于检索信号做兜底升级。
即使一个问题看起来像简单问题，我也不会完全相信预判。我会先跑一轮轻量召回，然后看几个信号：
一个是 Top-K 结果的相似度分布是不是足够集中；
一个是前几条结果是不是来自比较一致的资料范围，比如同一篇论文同一章节附近；
再一个是候选片段之间有没有明显冲突，或者相关片段太分散。
如果这些信号不好，比如前几条分数接近但都不高、结果跨很多文档而且主题发散，我就会自动升级到重链路，加 rerank 或更严格的元数据过滤。
所以它不是‘先验一次定生死’，而是‘先粗判，再根据召回质量决定要不要加重链路’。

如果只说我真实会落地的规则，我觉得最实用的就是这几条：
问题类型 + 是否要求引用 + 首轮召回质量，这三个信号已经够用了。因为规则太多会让系统难调，最后自己都解释不清为什么这次走轻、下次走重。

至于怎么避免用户觉得‘同样的问题有时快有时慢、有时准有时不准’，我觉得关键不是让所有问题都绝对一致，而是要让体验上有预期、可解释、结果保底。

第一，我会尽量保证默认回答风格一致。
不管走轻链路还是重链路，最终输出的结构尽量统一，比如都给‘简短回答 + 依据来源’这样的格式。这样用户感知到的是同一个产品在回答，只是有些问题会多花一点时间去做更严谨的检索，而不是两个完全不同的模式。

第二，我会做升级式体验提示，但提示要很轻。
比如系统检测到问题复杂时，可以在生成前给一个很轻的状态反馈，像‘正在补充检索依据’或者‘正在核对引用位置’。这样用户会理解为什么这次稍慢一点，是因为系统在做更严格的校验，而不是性能不稳定。

第三，我会做结果保底策略。
轻链路不是说为了快就可以乱答。只要轻链路下的检索置信度不够，我宁可少回答一点，或者明确说‘当前依据不够充分，建议查看以下相关片段’，也不要给一个看起来流畅但不可靠的答案。这样用户会觉得系统在不同速度下，至少可信度标准是一致的。

第四，我会把分流策略尽量设计成稳定命中而不是频繁抖动。
比如同样是‘解释某个概念’这类问题，规则应尽量长期落在同一类链路，不要今天因为一个阈值浮动走轻，明天又走重。也就是说阈值要留缓冲区，宁可少一点‘极致优化’，也要减少体验抖动。

所以如果总结成一句话，我会说：
我会用‘问题类型 + 是否要求引用 + 首轮召回质量’来决定轻重链路，先做可解释的简单分流，再用召回质量做升级兜底；同时通过统一输出格式、轻提示和低置信度保守回答，避免用户感受到明显的体验割裂。”

# 7、 你刚才提到“首轮召回质量不好就自动升级到重链路”。在你这个科研问答助手里，你会用哪一个最核心的离线指标来定升级阈值（比如 Top-1/Top-3 命中、相似度分布、文档一致性等选一个你最信的），以及你会把升级率大概控制在什么范围内（比如 10%/30%/50%），为什么？

如果线上要决定是否从轻链路升级到重链路，我最信的不是 Top-3 命中这种离线 gold 指标，而是首轮召回的相似度分布，尤其看 Top-1 和后面几个候选有没有明显分数断层。因为这个信号在线可得，也最能反映系统当前有没有把握。如果前几条分数都很接近，我就认为不确定性高，应该升级到重链路。升级率上，我会控制在 20% 到 30%，大概 25% 左右。太低说明很多该升级的问题没被拦住，太高又会把平均时延拉得太高，分流就失去意义了。

# 8、你说用“Top-1 和后续候选的分数断层”来决定是否从轻链路升级到重链路。你当时会怎么用一周线上日志 + 少量人工标注把这个“断层阈值”校准出来：具体你会标注什么（标注口径是什么）、怎么从日志里抽样（抽哪些类型的 query）、最后怎么判断这个阈值是“有效的”而不是拍脑袋的？

我会先定义一个很具体的标注口径，不是抽象标‘该不该升级’，而是标‘首轮召回是否已经足够支持正确且可追溯的回答’，再补一个‘gold 支持片段是否进了前 3’。抽样上我不会纯随机，而会按 query 类型和首轮分数形态分层抽样，重点多抽边界题。然后用一周日志去算 Top-1 和后续候选的分数 gap，看 gap 变小时，轻链路可用率和引用正确率是不是明显下降。如果这种关系稳定存在，再选一个同时满足升级率可控、升级后收益明显的阈值。这样这个阈值就是数据校出来的，不是拍脑袋定的。

原回答：
如果让我用 一周线上日志 + 少量人工标注 去把这个‘分数断层阈值’校准出来，我会把这件事拆成三步：先定义标注口径，再做有代表性的抽样，最后看这个阈值能不能同时改善效果和控制升级率，而不是只看单一指标。这个思路其实和我当时做 RAG 问答助手时一贯的做法一致，就是先把检索、引用和可用性分开看，再决定链路怎么调。

第一步，我会标注什么。
我不会一上来就标一个很抽象的‘这题该不该升级’，因为那样主观性太强。
我会对每条 query 先保留轻链路首轮召回的 Top-K 结果，然后人工标三类信息：

1、第一类是 首轮召回是否已经足够回答。
标注口径是：如果只看轻链路首轮召回出来的前几段，不加 rerank，是否已经能支持一个正确、基本完整、且能给出可靠出处的答案。这个我会标成 yes / no。
2、第二类是 gold 支持片段是否已经在前 3 条里。
这里的口径不是要求答案一字不差，而是看前 3 个候选里，有没有至少一个片段能作为这个问题的主要证据。这个标注会比较接近我们之前说的 Top-3 命中，只不过是在线日志回放场景下的人标版本。
3、第三类是 问题复杂度类型。
我会给 query 打一个轻量标签，比如事实定位、定义解释、对比总结、跨文档归纳、强引用需求这几类。因为我不希望最后阈值是被某一类 query 带偏的，我需要知道它在不同类型下表现怎么样。

如果资源再少一点，我至少会标前两类。因为本质上我最关心的是：轻链路在这条 query 上是不是已经够用。

第二步，怎么从日志里抽样。
我不会随机全抽，因为随机抽样很容易被大量简单 query 稀释，最后把阈值校得很乐观。
我会做一个分层抽样：

先按 query 类型分层，也就是刚才说的事实定位、解释类、对比总结类、强引用类。
再按首轮召回分数形态分层，比如：
（1）Top-1 明显高于后面候选的
（2）Top-1 和 Top-2 / Top-3 很接近的
（3）整体分数都偏低的
这样抽样的目的是把‘明显容易题’、‘边界题’和‘高风险题’都覆盖到。
因为这个阈值真正有价值的地方，不是在特别容易的题上，而是在那些看起来有点像能答、但其实很容易答偏的边界题上。

如果一周日志量不大，我会控制在一个比较现实的规模，比如 100 到 150 条 query 做人工标注。
这个规模不算大，但只要分层做得好，已经足够拿来校一个启发式阈值。
我的经验是，这种规则校准一开始不需要追求统计学上特别漂亮，关键是样本结构要对。

第三步，怎么从这些数据里校阈值。
我会先从日志里算每条 query 的一个核心特征：
比如 gap = score(top1) - score(top3平均)，或者更简单一点 score(top1) - score(top2)。
然后把样本按 gap 大小分桶，比如从很小到很大分成几档，看每一档里：

轻链路直接可用的比例是多少
gold 支持片段进入前 3 的比例是多少
如果这些题不升级，人工评估里出错率有多高

如果我发现当 gap 小于某个值时，‘轻链路已足够回答’的比例明显下降，或者引用错误率明显上升，那这个区间就可以作为升级阈值候选。
也就是说，我不是先定阈值再解释，而是先看 gap 和最终可用性之间有没有稳定关系。

最后，怎么判断这个阈值是有效的，而不是拍脑袋。
我会看三个结果：
第一，看它能不能把高风险 query 真正筛出来。
比如阈值以下的 query，是不是明显更容易出现‘首轮召回不够回答’或者‘引用不稳’的问题。
如果阈值两边没有明显差异，那说明这个特征不够好。
第二，看它带来的升级率是不是在可控范围。
比如我目标是控制在 20% 到 30% 左右，那如果某个阈值把 60% 的 query 都送进重链路，哪怕效果有提升，我也会觉得它不够实用。
反过来，如果只升级 5%，那通常又说明它太保守，抓不到真正的边界题。
第三，看它上线回放后，被升级那部分 query 的收益是否显著高于未升级部分。
换句话说，我希望看到的是：
被阈值判定去重链路的那些题，rerank 之后的可用性提升明显；
而留在轻链路里的那些题，本来就已经够好了。
只有这样，才能说明这个阈值真的起到了‘把重资源用在高风险问题上’的作用。
